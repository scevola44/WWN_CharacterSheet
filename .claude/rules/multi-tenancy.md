# Multi-Tenancy via Campaigns

This file is **not** WWN game rules. It documents the app's tenancy model — how characters and homebrew content are scoped to groups of players ("campaigns") so the same deployment can host many independent home games.

It is the locked design output of Phase 0 of the multi-tenancy rollout (see the top-level phased plan). Phases 1–4 implement what's described here. **When this file and the top-level plan disagree, this file wins** for schema and policy questions; the plan wins for sequencing.

---

## Scope and non-goals

**In scope**: Campaigns as a tenant boundary; per-campaign membership (GameMaster vs Player); per-campaign authoring of `FocusDefinition` / `Spell` / `Art` / `ArtSource`; visibility of characters across members; invite-code joining; cascade behaviour when members leave or campaigns are deleted.

**Not in scope**:
- Workspaces / organizations / Discord-style servers above the campaign.
- Cross-campaign content sharing or a marketplace.
- Per-character "share with party" opt-in (deferred to Phase 5).
- Fine-grained per-content ACLs (e.g., "only show this NPC to player X"). The visibility model is binary per role.

---

## Core concepts

### Campaign

A **Campaign** is the unit of tenancy. Every characters-vs-players grouping in the app is a Campaign. There is no superclass and no subclass.

A user may belong to many campaigns. A user owning a personal character with no campaign is also valid.

### Membership and roles

A **CampaignMembership** records that a user is a member of a campaign with one of two roles:

- **`GameMaster`** — administers the campaign: edits campaign metadata, authors campaign-scoped content, can read all `PlayerCharacter` rows in the campaign, can author `GmNpc` rows, can invite/remove members, can transfer GM, can delete the campaign.
- **`Player`** — has access to their own characters in the campaign and to campaign-scoped content for browsing.

A campaign always has at least one `GameMaster`. The last GM cannot leave without first transferring the role to another member (or deleting the campaign).

There is no global `GameMaster` role. The only global Identity role remains `Admin`, which is repurposed for operational duties (system content authoring, abuse handling, support tooling).

### Character roles within a campaign

A `Character.CharacterRole` field distinguishes player-facing characters from GM-private NPCs:

- **`PlayerCharacter`** — visible to its owner always; visible read-only to GMs of the character's campaign; invisible to other players.
- **`GmNpc`** — visible only to GMs of the character's campaign. Invisible to all players including the campaign's players.

`GmNpc` requires `CampaignId IS NOT NULL`. Personal characters (no campaign) are always `PlayerCharacter`.

### Invites

A **CampaignInvite** is a redeemable code generated by a GM. It is shareable as a link (no email infra in early phases — Phase 4 adds optional email-delivered invites).

### `Campaign` aggregate root

```csharp
// src/WWN.Domain/Aggregates/Campaign.cs
public sealed class Campaign
{
    public Guid Id { get; private set; }
    public string Name { get; private set; }
    public string? Description { get; private set; }
    public string OwnerUserId { get; private set; }       // creator; immutable, used for audit
    public DateTimeOffset CreatedAt { get; private set; }
    public DateTimeOffset UpdatedAt { get; private set; }

    private readonly List<CampaignMembership> _memberships = new();
    public IReadOnlyList<CampaignMembership> Memberships => _memberships;

    public static Campaign Create(string name, string ownerUserId, string? description = null);

    public void Rename(string name, string? description);
    public void AddMember(string userId, CampaignRole role);
    public void RemoveMember(string userId);              // throws if removing the last GameMaster
    public void ChangeRole(string userId, CampaignRole newRole);  // throws if it would leave zero GMs
    public void TransferGameMaster(string fromUserId, string toUserId);  // toUserId must already be a member
}
```

### `CampaignMembership` (entity within `Campaign` aggregate)

```csharp
// src/WWN.Domain/Aggregates/CampaignMembership.cs   (or under Campaign.cs)
public sealed class CampaignMembership
{
    public Guid CampaignId { get; private set; }
    public string UserId { get; private set; }            // AppUser.Id (string per ASP.NET Identity)
    public CampaignRole Role { get; private set; }
    public DateTimeOffset JoinedAt { get; private set; }

    internal CampaignMembership(Guid campaignId, string userId, CampaignRole role);
    internal void SetRole(CampaignRole role);
}

public enum CampaignRole { GameMaster, Player }
```

Composite primary key: `(CampaignId, UserId)`.

Mutation only goes through `Campaign` methods. The membership repository is a **query** convenience (`ListForUser(userId)`) and is allowed to read memberships independently, but writes always flow through the aggregate.

### `CampaignInvite` (its own entity, not part of the Campaign aggregate)

```csharp
public sealed class CampaignInvite
{

    public Guid CampaignId { get; private set; }
    public string CreatedByUserId { get; private set; }

    public int? MaxUses { get; private set; }                // null = unlimited; default null
    public int Uses { get; private set; }
    public DateTimeOffset? RevokedAt { get; private set; }   // soft-revoke; preserves audit trail

    public static CampaignInvite Create(Guid campaignId, string creatorUserId, DateTimeOffset? expiresAt, int? maxUses);
    public void Revoke();
    internal void RecordUse();                             // bumps Uses; thrown if expired/revoked/exhausted
}
```

Kept outside the `Campaign` aggregate because invites are append-mostly, may outlive the campaign briefly during rollback windows, and are queried by code (a unique index lookup) far more than by campaign.

## Visibility matrix

`X` = visible read-write; `R` = visible read-only; `—` = invisible (404 on direct access; absent from list responses).

### Characters

| Viewer relationship to character | `PlayerCharacter` | `GmNpc` |
|---|---|---|
| Owner of the character | X | X |
| GameMaster of the character's campaign (not owner) | R | X |
| Other Player in the character's campaign | — | — |
| Member of a different campaign | — | — |
| Not a member of any related campaign | — | — |
| Personal character (no campaign), viewer = owner | X | n/a (invariant: personal ⇒ PlayerCharacter) |
| Personal character, viewer ≠ owner | — | n/a |

Notes:
- "GM sees PlayerCharacter as R" means the API returns the full character DTO with a `canEdit: false` flag; UI renders the sheet in a non-mutating mode.
- Direct access to a hidden character returns **403 Forbidden** if the route would otherwise resolve (the requester *could have* known the ID), and **404 Not Found** if the ID does not exist or is structurally hidden (e.g., a player asking for an NPC ID they could never have learned). Phase 2 acceptance specifies `403` for the "can otherwise see the character" case to avoid leaking existence.

### Content (`FocusDefinition`, `Spell`, `Art`, `ArtSource`)

| Content `CampaignId` | Viewer = Admin | Viewer is GM of that campaign | Viewer is Player of that campaign | Viewer is in another campaign | Viewer has no related campaign |
|---|---|---|---|---|---|
| `null` (system) | X | R | R | R | R |
| Set to campaign C, viewer matches C | X | X | R | — | — |
| Set to campaign C, viewer not in C | X | — | — | — | — |

System-content writes are gated to `Admin`. Campaign-content writes are gated to `GameMaster` of that campaign. Read of campaign content requires membership of that campaign **or** that the campaign is the viewer's "active campaign" filter (which the backend additionally validates is a campaign the viewer belongs to).

## Active-campaign mechanism

The frontend owns an "active campaign" concept. It is communicated to the backend as an explicit `?campaignId=` query parameter on content-browsing endpoints. The backend is stateless about which campaign is active; it only validates that the requester is a member.

- Frontend stores the active campaign in React context, persisted to `localStorage`. Survives page reload; per-tab if we ever want it (today: shared via storage events).
- Active campaign may be `null` ("no campaign / personal / system"). Content-browse endpoints with no `?campaignId=` return system content only.
- Sheet-level routes (`/api/characters/{id}/...`) **do not** consult active campaign; access is computed from the character's own `CampaignId` and the requester's memberships.
- Character-create UI may default a new character's `CampaignId` to the active campaign, but the backend treats `campaignId` on the create payload as authoritative.

## Authorization policy interfaces

Sketch only. Phase 2 implements them under `WWN.Application/Authorization/` (or a similar new folder).

```csharp
public interface ICharacterAccessPolicy
{
    bool CanRead(ClaimsPrincipal user, Character character, IReadOnlyCollection<CampaignMembership> userMemberships);
    bool CanWrite(ClaimsPrincipal user, Character character, IReadOnlyCollection<CampaignMembership> userMemberships);

    // Used by repositories to materialize a SQL filter without loading all rows.
    // Returns a predicate suitable for IQueryable<Character>.Where(...).
    Expression<Func<Character, bool>> ReadableFilter(string userId, IReadOnlyCollection<CampaignMembership> userMemberships);
}

public interface ICampaignAccessPolicy
{
    bool IsMember(string userId, Guid campaignId, IReadOnlyCollection<CampaignMembership> userMemberships);
    bool IsGameMaster(string userId, Guid campaignId, IReadOnlyCollection<CampaignMembership> userMemberships);
}

public interface IContentAccessPolicy
{
    // Single overload per content type kept off this sketch for brevity.
    bool CanRead(ClaimsPrincipal user, Guid? contentCampaignId, IReadOnlyCollection<CampaignMembership> userMemberships);
    bool CanWrite(ClaimsPrincipal user, Guid? contentCampaignId, IReadOnlyCollection<CampaignMembership> userMemberships);
    Expression<Func<T, bool>> ReadableFilter<T>(string userId, Guid? activeCampaignId, IReadOnlyCollection<CampaignMembership> userMemberships) where T : class;
    // T must expose Guid? CampaignId — enforced by a marker interface or by convention in the impl.
}
```

Implementations are **pure functions** of the inputs; no DB access in the policy itself. The Application layer loads the user's memberships once per request (cached in a scoped service) and passes them to the policy.

ASP.NET authorization policies (`CampaignMember`, `CampaignGameMaster`) wrap these and handle the `[Authorize(Policy = "...")]` attribute path.

## Lifecycle rules

### Member leaves a campaign

`CampaignService.RemoveMember(campaignId, userId)`:
1. Load `Campaign`. Call `Campaign.RemoveMember(userId)` — throws if `userId` is the last `GameMaster`.
2. Find all `Character` rows where `CampaignId = campaignId AND OwnerUserId = userId`. For each: `character.LeaveCampaign()` (sets `CampaignId = null`, role → `PlayerCharacter`).
3. Save in a single transaction.

Effect: GM stops seeing those characters from the next request onward.

### Campaign deleted

`CampaignService.DeleteCampaign(campaignId)`:
1. Verify caller is a `GameMaster` of the campaign (or `Admin`).
2. For each member's characters in this campaign, call `character.LeaveCampaign()`.
3. For each `GmNpc` in this campaign: change role to `PlayerCharacter`, set `CampaignId = null`. Owner is unchanged (the original GM-author keeps them as personal PCs).
4. Soft-revoke all open invites (or hard-delete; either is fine since the campaign is going).
5. Delete the `Campaign` aggregate (cascades memberships).
6. All in one transaction.

### GM transfers role

`Campaign.TransferGameMaster(fromUserId, toUserId)`:
- `toUserId` must already be a member.
- After: `to` has `Role = GameMaster`, `from` has `Role = Player`. (Multiple GMs are permitted; if a campaign already has multiple GMs, the source can simply leave instead.)
- **NPC ownership is unchanged.** The original GM still owns their NPCs. The new GM gains write access via the policy. If the original GM later leaves the campaign, their NPCs orphan to them as personal `PlayerCharacter` rows (rule above).

### Account deleted

Out of scope for Phases 0–3; defined in Phase 4. Sketch:
- If the user is the sole GM of any campaign, account-delete is rejected with a directive to transfer or delete those campaigns first.
- Otherwise, on delete: the user's characters are hard-deleted (or soft-deleted per Phase 4 policy); their memberships are removed; their authored content has `OwnerUserId` blanked (still owned by the campaign).

## Invite codes

- **Alphabet**: lowercase a–z plus 2–9 (no `0`, `1`, `o`, `l`, `i` — ambiguous when shared by voice or written by hand).
- **Length**: 12 characters. ~62 bits of entropy; sufficient given Phase 4 rate-limiting on the redemption endpoint.
- **Default expiry**: `now + 7 days`. Configurable per invite at creation time. Setting `null` (never expires) is allowed but discouraged; surface a warning in the GM UI.
- **Default `MaxUses`**: `null` (unlimited). GM may set a finite cap.
- **Revocation**: soft (`RevokedAt` timestamp). Audit-friendly. Revoked invites cannot be redeemed even if not expired and within use-cap.
- **Redemption flow** (`POST /api/campaigns/join/{code}`):
  1. Look up by `Code`. 404 if absent.
  2. Reject if revoked / expired / `Uses >= MaxUses`.
  3. If the requester is already a member of the campaign: no-op success, return current membership. (Avoids "invite already used by you" UX confusion when the GM shares the link in multiple places.)
  4. Otherwise: add membership as `Player`, increment `Uses`, return the new membership and the campaign summary.
- **Generation collision**: re-roll on insert if the unique index trips. With 12 chars from a 32-char alphabet, collisions are negligible at any realistic scale.

## Migration plan summary

All new columns on existing tables are nullable or have safe defaults. No data backfill is required; existing rows survive untouched.

| Phase | Migration | Adds |
|---|---|---|
| 1 | `AddCampaigns` | `Campaigns`, `CampaignMemberships`, `CampaignInvites` tables. No changes to existing tables. |
| 2 | `AddCharacterCampaign` | `Characters.CampaignId` (nullable FK), `Characters.CharacterRole` (int, default `0` = `PlayerCharacter`). |
| 3 | `AddCampaignIdToContent` | `CampaignId` (nullable FK) on `FocusDefinitions`, `Spells`, `Arts`, `ArtSources`. Existing rows stay `NULL` = system content. |
| 4 | `AddAuditLogAndUserFlags` | `AuditLogEntries` table; `AppUsers.SuspendedAt` (nullable). Out of scope here; documented for migration ordering. |

Migrations must land in order. Skipping is not supported.

## Cross-cutting implementation notes

- **EF Core change tracking**: when `Character.JoinCampaign` / `LeaveCampaign` mutates `CampaignId`, also touch `UpdatedAt` if/when that field is added; do not rely on EF auto-detecting Guid? changes via property fixup alone in deeply navigated graphs.
- **AuthContext extension** (Phase 1+): a single fetch at app load returns `{ user, campaignMemberships: CampaignMembership[] }`. Memoize the array; consumers should use a `useActiveCampaign()` hook for the `?campaignId=` value.
- **403 vs 404**: prefer `403` whenever the requester could plausibly have learned the ID (e.g., it's their own former character). Use `404` only when the ID is fully unknowable to them.
- **Endpoint authorization is non-negotiable**: every endpoint in scope of this work gets `RequireAuthorization(...)` with the appropriate policy. The current `FocusDefinitionEndpoints` / `SpellEndpoints` / `ArtEndpoints` lack of authorization is a known gap closed in Phase 3.

## Open questions deferred to later phases

- **`VisibleToParty` per character** — Phase 5. Until then, players never see each other's sheets.
- **Email-delivered invites** — Phase 4 (when `IEmailSender` exists).
- **Cross-campaign content sharing / marketplace** — out of roadmap. Would require a new "Published" flag and moderation policy.
- **Per-content ACLs** (e.g., share one spell with one player) — out of scope; the model is binary by membership.
- **Audit trail of role changes / invite revocations** — Phase 4 audit log.
- **2FA for `Admin`** — Phase 5.
- **Soft-delete window for campaigns** (undo within N days) — not in current scope; campaign delete is hard with NPC orphaning.
